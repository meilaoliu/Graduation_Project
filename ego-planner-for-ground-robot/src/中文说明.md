# Environment Configuration

## qpOASES

We use qpOASES to solve MPC，and MPC is used for trajectory tracking

qpOASES  solver installation :

```bash
cd qpOASES
mkdir build
cd build
cmake ..
sudo make
sudo make install
```

## patchwork

patchwork is used for ground segmentation. This planner accepts 3d point cloud information **Pointcloud2**. After the point cloud is segmented on the ground, the point cloud is projected to the 2d plane to construct a 2d grid map for planning.

Follow the official repository to download this package：[patchworl_github](https://github.com/LimHyungTae/patchwork)

## Simulation environment

The simulation environment uses the open source simulation environment of the CMU Robotics Institute. Users  need to install the relevant libraries and download the mesh file. We recommend to use the indoor environment to test the planner.

The environment have two versions for ros-melodic and ros-noetic. The package in this project is for ros-melodic.

[CMU-environment](https://www.cmu-exploration.com/)

若提示缺少 `openai` Python 包，可在当前 ROS Python 环境中安装：
```bash
pip install openai
```

`nlp_commander_node.py` 是早期单文件版本，内部坐标和任务逻辑已经不再与当前变电站 OSM 地图保持同步；后续实验建议统一使用 `substation_nlp_commander_node_v2.py`。

关于模型选择，阿里云百炼文档推荐通用任务优先使用 `qwen3.6-plus`。
它支持 OpenAI 兼容 Chat Completions、JSON Mode、Function Calling 和 100 万上下文，足够覆盖当前“中文指令 → 结构化巡检任务”的场景。
如果后续要处理更复杂的多轮推理、跨文档规划或很绕的组合指令，可以把 `DASHSCOPE_MODEL` 改为 `qwen3.6-max-preview`。
注意模型 ID 不是 `qwen 3.6 max`，而是 `qwen3.6-max-preview`；该模型推理能力更强，但成本更高、上下文为 256k，当前不建议作为默认值。
Qwen3.6 系列默认支持思考模式，在线机器人控制更看重响应速度，所以代码默认通过 `DASHSCOPE_ENABLE_THINKING=false` 关闭；需要提升复杂指令理解时再改为 `true`。
若要单独测试百炼接口，可在确认 API Key 有效后运行 `RUN_REAL_LLM_TEST=1 python3 src/nlp_commander/tests/test_llm_simple.py`；默认不会主动发起真实请求。
如果测试输出 `Connection error`，先确认 `DASHSCOPE_BASE_URL` 没有被误设，并运行以下不含密钥的连通性检查：
```bash
curl -sS -o /dev/null -w '%{http_code} %{remote_ip} %{ssl_verify_result}\n' \
  https://dashscope.aliyuncs.com/compatible-mode/v1/models
```
正常情况下，该命令应能返回 `401` 和远端 IP，表示网络、DNS 与 TLS 都是通的；随后再检查测试脚本打印出的 `base_url`、`key_length` 和 `proxy_env`。
如果报错中出现 `Unknown scheme for proxy URL URL('socks://127.0.0.1:7890/')`，说明 Python 的 `httpx` 不接受当前 `socks://` 代理写法。
最简单的处理方式是临时忽略代理后重试：
```bash
export DASHSCOPE_CLEAR_PROXY=1
RUN_REAL_LLM_TEST=1 python3 src/nlp_commander/tests/test_llm_simple.py
```
若必须使用代理，则优先把代理变量改成 Clash 常见的 HTTP 代理地址：
```bash
export http_proxy="http://127.0.0.1:7890"
export https_proxy="http://127.0.0.1:7890"
export all_proxy="http://127.0.0.1:7890"
```
只有确认当前 Python 环境安装了 SOCKS 支持时，才使用 `socks5://127.0.0.1:7890`，不要写成 `socks://127.0.0.1:7890/`。

### LLM语义指挥节点说明

`nlp_commander` 的 V2 版本对应论文第二章中的前端任务规划流程：

1. `maps/simplified_substation.osm` 作为简化语义地图注入大模型提示词，大模型输出 `task_type`、`targets`、`execution` 和 `route_policy` 等 JSON 字段。
2. `maps/substation.osm` 保存完整节点坐标与 `<way>` 连通关系，代码优先从 OSM 中读取拓扑边；若地图缺少连通关系，才回退到内置邻接表。
3. `IntentNormalizer` 对大模型 JSON 做字段补全、类型转换和目标合法性校验；具体语义如重复次数、限时、返航、停留和拍照要求，优先来自 JSON，而不是主节点里的关键词规则。
4. `PathPlanner` 使用 Dijkstra 计算相邻目标之间的最短路径。多目标无显式顺序时先按变电站区域成组，再按拓扑距离选择区域入口和区域内连续访问顺序；用户明确给出顺序时保留该顺序。
5. 全局轨迹模式下，`SegmentScheduler` 只在 `targets[].stop_required=true` 的目标停留，并只在 `photo_required=true` 时拍照；同一设备若只是作为后续路径的途经点，不会重复停留和重复拍照。
6. `RuntimeEventLog` 记录任务执行过程中的段开始、段完成、失败、低电量、返航充电等事件；主节点还会在任务结束或限时结束时生成一份“上一任务摘要”，把用户原始指令、目标范围、执行约束、已完成点、异常和结束状态压缩后注入下一轮提示词。
7. 低电量返航不再直接给充电口局部目标，而是先在变电站拓扑图上用 Dijkstra 规划到入口点的全局路径；充满后恢复任务时，也会从入口点重新接入下一段拓扑路径。
8. `BatteryPolicy` 会在执行下一段前计算“当前位置到下一目标 + 下一目标到充电点 + 安全余量”的所需续航。当前版本优先使用 `/battery_state.estimated_remaining_distance_m` 作为可用续航估计，与仪表盘“剩余可走”显示保持同源；没有该字段时，才按 `battery_full_range_m` 做备用换算。
9. 限时巡检不再把 `duration_minutes` 预估成多遍重复路径。系统先执行一轮目标序列，并在每段事件中记录已用时间、剩余时间；一轮结束后若仍有时间，会从当前位置按拓扑代价就近生成下一批候选目标，再追加剩余时间内可完成的完整段或前缀段。
10. `TaskAgentRuntime` 在段失败、进度异常、限时续巡结束或调度器明确请求决策等关键事件处启动自主决策流程。大模型需要先调用本地只读工具读取状态、规划路径并估算电量/时间，再输出离散任务动作；动作执行前统一经过 `SafetyValidator` 校验。
11. `WaypointManager` 是旧版逐航点模式，将规划出的 Waypoint 序列逐点发布到 `/move_base_simple/goal`，并通过 `/odom_adjust` 判断是否到达当前目标。

V2 节点期望大模型返回如下结构化意图：
```json
{
   "task_type": "area_inspection",
   "targets": [
      {"name": "高压配电区巡检点1", "priority": 1, "stop_required": true, "photo_required": true}
   ],
   "execution": {"repeat_count": 3, "duration_minutes": null, "until_time": null, "preserve_order": false},
   "route_policy": {"use_topology": true, "optimize_order": true, "return_home_on_low_battery": true}
}
```
例如“去高压区反复巡检三遍”应由大模型在 `execution.repeat_count` 中给出 `3`；“巡检20分钟”“巡检到15:30”分别进入 `duration_minutes` 和 `until_time`。
“回到入口点”“回到起点”应解析为 `task_type=return_home`，目标为 `入口点`，且 `photo_required=false`。
“返回充电”“去充电”“返航充电”应解析为 `task_type=go_charge`，目标同样为 `入口点`，但到达后会调用充电流程。
如果用户在同一句话中使用“先、然后、接着、最后”等顺序表达，或者包含多个不同目标范围/不同时间约束，模型应返回 `task_plan.stages` 多阶段结构：

```json
{
   "reasoning": "先复用上一任务范围，再巡检低压区，最后返航。",
   "task_plan": {
      "mode": "sequential",
      "stages": [
         {
            "stage_id": "stage_1",
            "task_type": "area_inspection",
            "targets": [
               {"name": "高压配电区巡检点1", "priority": 1, "stop_required": true, "photo_required": true}
            ],
            "task_description": "沿上一任务范围限时巡检五分钟",
            "execution": {"repeat_count": 1, "duration_minutes": 5, "until_time": null, "preserve_order": false},
            "route_policy": {"use_topology": true, "optimize_order": true, "return_home_on_low_battery": true}
         },
         {
            "stage_id": "stage_2",
            "task_type": "area_inspection",
            "targets": [
               {"name": "低压配电室1", "priority": 1, "stop_required": true, "photo_required": true}
            ],
            "task_description": "巡检低压区一圈",
            "execution": {"repeat_count": 1, "duration_minutes": null, "until_time": null, "preserve_order": false},

```
主节点会先执行第一个阶段，并在收到 `task_completed` 事件后自动启动下一阶段；每个阶段仍复用原有路径规划与分段调度逻辑。

# 参数调整说明

在 CMU 提供的仿真环境中，小车并不具备真实的物理引擎，能够完全按照控制指令 (线速度 v 和角速度 w) 进行运动，这会影响到 MPC 控制器的参数设定

MPC 控制器的目标函数由两部分组成：跟踪误差和运动限幅。用户可以通过下面 4 个参数进行调整：

```xml
<!-- adjust in advanced_param.xml -->
<param name="MPC/v_max" value="1.8" type="double"/>
<param name="MPC/w_max" value="1.2" type="double"/>
<param name="MPC/omega0" value="1.0" type="double"/>
<param name="MPC/omega1" value="0.5" type="double"/>
```

`MPC/v_max` 与 `MPC/w_max` 与规划时设定的最大速度与加速度有关，可以设置成略大于规划最大速度 `max_vel` 的数值

`MPC/omega0` 和 `MPC/omega1` 分别为上述跟踪误差和运动限幅两项的权重：

1. `omega0` 越大，MPC 求解结果越倾向于减小误差，但有可能超过机器人的运动学极限
2. `omega1` 越大，MPC 求解结果越倾向于使得机器人的运动更平滑，但与参考轨迹之间的误差会增大

在 CMU 提供的仿真环境中，本人建议调大 `omega0`，因为该仿真环境中小车的运动不受限制

但在存在物理引擎的仿真环境中，建议适当调大 `omega1`，否则很容易出现角速度和线速度过大的现象，导致机器人运动不平滑

计划在分支中提供一版联合 SLAM 和具有物理引擎的仿真环境的 ego-planner，后面有时间整理后再上传

# 智能巡检系统（毕设联调）

> 本节描述在原 ego-planner-for-ground-robot 之上新增的"自然语言指令 → 分段全局轨迹 → 拍照 → 低电返航"完整巡检流水线。
> 全部新功能与 **B 样条 / MINCO** 两种轨迹后端均兼容（仅依赖高层 `planGlobalTrajWaypoints` 与 `/global_waypoints` 接口）。

## 系统组成

| 包                        | 节点 / 内容                                    | 作用                                                              |
| ------------------------- | ---------------------------------------------- | ----------------------------------------------------------------- |
| `nlp_commander`           | `substation_nlp_commander_v2` + `SegmentScheduler` | LLM 解析指令 → Dijkstra 路径 → 按"停留点"切段 → 调度全局轨迹       |
| `inspection_services`     | `photo_service_node` + `/take_photo`           | 缓存 `/camera/image`，按需保存照片并广播 `/photo_event`           |
| `battery_simulator`       | `battery_monitor_node`                         | 虚拟电量：基础时间衰减 + 运动消耗 + 拍照消耗，发布 `/battery_state` 与 `/low_battery_alert` |
| `inspection_dashboard`    | `dashboard_node.py` (Flask + SocketIO)         | 浏览器仪表盘：状态 / 实时画面 / 对话框 / 拍照画廊                 |
| `ego_planner` (修改)      | FSM 新增 `flight_type=3` (DYNAMIC_WAYPOINTS)   | 订阅 `/global_waypoints`，分段执行；段完成时广播 `/segment_done`   |

数据流：

```
NL 指令 ─┐
浏览器 ─┴→ nlp_commander.SegmentScheduler ──/global_waypoints──▶ ego_planner FSM
                  ▲                              │
                  │                              └──/segment_done──┐
                  ├──/low_battery_alert────── battery_monitor      │
                  ├──/take_photo (srv) ────── photo_service ────/photo_event──┐
                  └──/charge   (srv) ──────── battery_monitor                 │
                                                                              ▼
                                                                inspection_dashboard
```

```bash
catkin_make             # 首次或修改 C++/包结构后

    realistic:=false           \   # true = 高保真场景 (慢)

# 浏览器打开 http://localhost:5000/
```

若要显式开启并测试任务级 Agent Runtime，可使用：

```bash
roslaunch inspection_dashboard inspection_full.launch \
   realistic:=false \
   use_global_traj:=true \
   enable_agent_runtime:=true \
   agent_min_tool_calls_before_final:=2 \
   agent_min_confidence:=0.35 \
   dashboard_port:=5000
```

若要专门测试“段失败后自动重试/二次失败跳过”，可把段完成超时临时调小：

```bash
roslaunch inspection_dashboard inspection_full.launch \
   realistic:=false \
   use_global_traj:=true \
   enable_agent_runtime:=true \
   segment_done_timeout:=3.0 \
   agent_max_retry_per_target:=1
```

正常仿真不建议长期使用 `segment_done_timeout:=3.0`。
当前架构中，固定超时只是 watchdog 时间窗；若到达该时间窗时里程计仍显示机器人存在位移或沿当前段参考路径取得进展，调度器会延长等待。真正的异常判断主要由 `SegmentProgressMonitor` 根据里程计进度异步完成，并通过 `progress_anomaly` 事件交给 Agent Runtime。

切换规划器后端：在 `ego-planner/planner/plan_manage/launch/advanced_substation_param.xml` 中
修改 `manager/use_minco`：`true` = MINCO，`false` = B 样条。两种后端下的接口完全一致。

## 段切分规则

`SegmentScheduler` 按 `SubstationGraph` 输出的 `(name, x, y, stop_required, photo_required)` 列表切段：
* `stop_required=True` 的航点是段终点；`False` 的航点是段内 min-snap 中间约束（不停车）
* `photo_required=True` 的段终点才调用 `/take_photo`；返回入口点或只需停车不拍照的目标可设为 `False`
* 起始位置（机器人当前位姿）作为第一段的起点；若当前位置只是路径起点，不会额外停留拍照；若当前位置本身也是本轮巡检目标，则会保留为一个零距离停留段并执行拍照/停留
* 每段终点速度 / 加速度 = 0（与论文一致）
* 对 `duration_minutes`、`until_time` 这类限时任务，只切首轮目标段；后续是否继续巡检由运行时剩余时间决定。续巡时会从当前位置重新按拓扑距离排序候选目标；若剩余时间不足以完成完整下一批，但还能完成前几个目标，调度器会追加可完成的前缀段，避免一开始机械展开几十个重复段

## 任务反馈、Agent Runtime 与预测性充电

`SegmentScheduler` 在任务执行中持续维护一份运行时上下文，内容包括：
* 当前段目标、段类型、剩余段数量
* 最近的段开始/完成/失败事件
* 任务已用时间、剩余时间和当前限时巡检轮次
* 电量百分比、低电量告警、是否正在返航充电
* 充电后从入口点恢复到下一任务段的拓扑路径

这份上下文会进入后续 LLM 指令解析的 Prompt，使系统具备论文第二章所描述的“执行反馈注入大模型推理”的基础能力。
任务结束或限时自然结束后，主节点会保留一份更短的上一任务摘要。
当用户说“刚才、上一轮、继续、再按之前、同样要求”等省略表达时，大模型会优先参考该摘要和近期反馈，而不是只复用上一条拓扑路线。
此外，`TaskAgentRuntime` 会消费这些运行时事件，并将事实、工具、校验和动作执行分开组织：

```text
WorldModel -> ToolRegistry -> AutonomousSupervisor -> SafetyValidator -> ActionExecutor
```

`WorldModel` 记录当前节点、电量、失败目标、剩余目标和近期事件。
`ToolRegistry` 提供只读工具，例如读取机器人状态、查询地图邻接、规划拓扑路径、估算距离、电量和时间。
`AutonomousSupervisor` 使用 ReAct JSON 循环工作：在输出最终动作前至少调用两次本地工具，不再默认把失败交给用户选择。
`SafetyValidator` 统一拦截危险动作，例如未知目标、拓扑不可达、低电量下继续普通巡检、同一目标超过重试次数等。
`ActionExecutor` 再把通过校验的动作映射到现有导航入口，例如重试失败目标、跳过目标继续剩余任务、按拓扑返航或结束任务。
Agent 输出的是离散任务级动作：`retry_failed_target`、`skip_target`、`reorder_remaining_targets`、`resume_remaining_task`、`return_home`、`go_charge`、`finish_task`、`abort`、`continue`。
对于限时任务，Agent 重试、跳过或恢复剩余目标时会继承原任务的绝对截止时间，不会因为重建任务而退化成普通巡检。
其中 `go_charge` 会调用 `SegmentScheduler.start_charge_task()` 启动独立返航充电段：机器人先按拓扑路径回到默认充电点（当前为入口点），随后执行原有 `/charge` 服务流程。
`return_home` 只表示返航到入口点，不会调用 `/charge`；只有 `go_charge` 或调度器插入的 `return_charge` 段才会触发充电。
`battery_monitor` 默认要求机器人位于充电点半径内才接受 `/charge`，避免在任意位置手动调用服务也能充电。
执行中异常不再只依赖固定段超时。
`SegmentProgressMonitor` 会以当前段折线路径为参考，持续计算路径投影进度、当前位置到路径的偏离量、到目标距离变化和最近位移。
当出现长时间无进展、疑似卡住、明显偏离路径或持续远离目标时，它只发出 `progress_anomaly` 事件。
该事件只表示“当前段可疑”，不会直接累计真实失败次数；只有 `segment_failed` 这类结果事件才会增加失败计数。
调度器会在发出 `progress_anomaly` 前先发布原地停止路径，让底层规划器先停住，再交给 Agent Runtime 决策。
当某段最终失败或超时时，调度器会先发布一条指向当前里程计位置的原地停止路径，再上报 `segment_failed` 事件给 Agent Runtime。
`return_charge` 返航充电段执行中的 `progress_anomaly` 只进入运行时上下文，不触发 Agent 抢占，避免正在返航时被重复 stop 和重启。
是否继续等待、重试当前目标、返航或跳过，仍由 `TaskAgentRuntime` 结合工具调用和安全校验决定。
不过，电量安全不交给大模型直接判断，而是由确定性策略先兜底。
因此 `energy_guard`、`battery_low` 和 `charge_completed` 默认只更新上下文，不触发 Agent 抢占执行；调度器会保留剩余巡检段，充电完成后按原任务续接。
`time_budget_finished` 和限时续巡不足导致的 `decision_required` 也只作为终止事实记录，不再触发新的返航/结束动作，避免任务完成后反复进入 Agent 循环。
如果 Agent 因进度异常或续航不足抢占执行 `go_charge`，调度器会停止当前段，但保留当前正在执行的 mission、后续待执行 mission 和限时上下文。
若充电完成时限时任务已经到时，则系统会明确输出限时结束并丢弃剩余巡检段，不再显示“恢复任务”。
预测性充电的判据为：

```text
当前电量可行驶距离 >= 当前位置到下一目标距离 + 下一目标到充电点距离 + 安全余量
```

若不满足该条件，系统会在执行下一目标前先插入返航充电段。
相关参数可在 `nlp_commander.launch` 中调整：
* `enable_predictive_charging`：是否开启预测性充电
* `battery_full_range_m`：满电估计可行驶距离，仅在 `/battery_state` 没有提供 `estimated_remaining_distance_m` 时作为备用换算参数
* `battery_reserve_m`：返航安全余量
* `time_budget_min_remaining_s`：限时任务剩余时间低于该值时不再追加新巡检段
* `time_budget_nominal_speed`：估算下一轮或部分续巡段耗时所用的名义速度
* `max_time_budget_cycles`：限时任务动态续巡的最大轮数
* `runtime_context_events`：保留多少条运行时反馈事件
* `enable_progress_monitor`：是否启用执行中进度监控
* `progress_monitor_no_progress_timeout_s`：路径进度长时间不增加时触发异常的时间阈值
* `progress_monitor_stuck_timeout_s`：机器人长时间几乎不动时触发异常的时间阈值
* `progress_monitor_path_deviation_m`：当前位置偏离当前段参考折线的距离阈值，默认 8m，且需伴随长时间无路径进展才触发
* `enable_agent_runtime`：是否启用任务级 Agent Runtime
* `agent_max_tool_iterations`：单次事件最多允许几轮工具调用/最终动作尝试
* `agent_min_tool_calls_before_final`：输出最终动作前至少调用多少个本地工具
* `agent_min_confidence`：Agent 最终动作的最低置信度
* `agent_max_retry_per_target`：同一失败目标允许自动重试的次数
* `agent_max_consecutive_failures`：连续失败达到该阈值后拒绝继续重试
* `agent_temperature`：Agent 决策阶段的大模型温度
* `agent_max_tokens`：Agent 决策阶段的大模型输出上限

## 设计取舍

* 分段调度走的是 ego_planner 现成的 `planGlobalTrajWaypoints()`（min-snap），与局部规划无关，因此 B 样条 / MINCO 切换不影响段调度。
* 低电量返航段会先在变电站拓扑图上规划到入口点的路径，再调用调度器，避免直接把充电点当局部目标。
* 电量、距离、可达性属于硬安全约束，优先由确定性算法处理；大模型主要负责语义理解、任务重组和面向用户的解释。
* `TaskAgentRuntime` 属于任务层智能体，不替代 ego-planner FSM，也不直接控制速度、角速度或轨迹控制点；它只决定重试、跳过、重排、返航、充电、结束或继续。
* 拍照通过 ROS service 同步执行，避免与导航并发竞争。
* 仪表盘后端不开 eventlet，使用 `async_mode='threading'` 避免 monkey-patch 与 rospy 冲突。
